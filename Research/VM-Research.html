2.10 v11 — hierarchical scope, not single-site (architecture revised)

Major correction

Earlier versions of this pack described v11 as "the single-site issues view" with weak evidence. That was a fundamental architectural misread. v11 is a hierarchical scoped issues view at any level of the org tree — estate root (handled by v10), regional aggregates (Asia / Europe / North America), and per-site pages (Zenith Corp, SecureNet, Nova Solutions, DataShield, CyberGuard Inc, Apex Digital, Gin & Juice Shop). Eleven HTML files simulate cross-page navigation. The case for v11 is therefore much stronger than I'd characterised — it directly implements the most strongly-evidenced enterprise-scoping demand in this pack.

What v11 actually is:

  • Left site-tree panel on every v11 page (340px wide, with filter + search). Shows all 172 sites organised in a 4-level tree: estate → region → sub-folder → site. A separate "Pentest group" classification sits alongside the geographic regions, hinting at orthogonal grouping (team / engagement-based, not just hierarchical).
  • Three regional aggregate pages — Asia (the default v11.html landing), Europe, North America. Each shows a folder header with edit-name + delete-folder + new-site actions, then folder tabs (Dashboard / Scans / Scheduled scans / Issues / Details), then the same SLA donuts + filter bar + 13-column issue table as v10. Cross-region tags (e.g. green "Europe" badge at #2E8802) appear on issues so you can see which sites in the current view belong to which regions.
  • Seven per-site pages — same shell, but with a back button + site-name title + delete-site + new-scan actions, a pre-scan check card with run-prescan button, site sub-tabs, and an embedded .issues-grid-card. The table adds a Path column (since site is implicit at this scope).
  • Cross-page navigation simulated in JavaScript: clicking a region in the tree navigates to the corresponding HTML file. 'Europe' → v11-europe.html, 'Asia' → v11.html, etc.

How this maps to the evidence:

  • Strong fit with Gong's "role-based dashboard scoping" (Section 2.19 below) — region-level views are precisely what leadership / regional managers would use; per-site views are precisely what the developer / app-owner audience would use. v11 is the implementation of the workflow Gong identified.
  • Strong fit with PM doc's "system of record" framing — hierarchical scope matches how enterprises actually organise (Visa / Mastercard / Amgen scale) and is compatible with the policy-by-criticality ask Amgen raised in PM Assertion 1.
  • Pentest group as a separate top-level grouping connects directly to the System C Healthcare ask in Section 2.9 (importing third-party pen-test results into Burp DAST as a managed-elsewhere category). The architecture is open-ended enough to support it.
  • Cross-region tags on issues are an interesting design choice — they admit that a "site" can belong to multiple groupings, which addresses some of the cross-deployment FP federation concern in Section 2.26 (validated FPs would carry their region tag with them).

The earlier-version concern that "no customer asked for a per-site issues page" was technically true but missed the bigger story: customers asked for scoped views by their org structure, and v11 delivers a scope-by-position pattern that flexes to any tree level. Per-site is just the leaf case.

2.13 Status / FP scope — substantially resolved in the bulk-action modals

Methodology correction

Earlier versions of this pack flagged scope-resolution as the highest-leverage UX gap in the design. That was wrong. v10's Edit issue status and Change severity bulk-action modals already include a scope radio group with three explicit options. I missed this because I sampled the table and side panel but didn't read the modals. Reviewing all of v10 (not just the parts I had a question about) corrected the picture. The remaining gap is narrower than I'd claimed.

What happens with an "issue" in DAST if I mark it as a false positive or as an accepted risk?… Does marking it in one site/place make the change for all sites?… How do I apply false positives and edit issue severity globally and or per site?
Carson · ARR Tier 1 · Case 00307715 · Aug 2025

What v10 actually does. When the user triggers Edit-status or Change-severity (bulk or per-row), a confirmation modal opens with three scope radios:

  • "This issue and all existing issues of the same type for this site" (default)
  • "This issue and all existing issues of the same type and URL for this site"
  • "These issues across all sites and future occurrences"

Plus a notes textarea. Plus an info note: "Issues found in future scans will be marked as the status according to the global setting". This directly answers Carson's "globally and or per site" question — the modal makes scope a first-class part of the action.

Where the gap remains: dynamic-URL patterns. The "same type and URL for this site" option marks a single URL as FP, not a URL pattern. AS Watson and Babiel describe URLs containing UUIDs (/users/anonymous/<uuid>) where every scan sees a fresh URL — none of the three radio options handles this cleanly. The current options are: too narrow (just one URL), too broad (all URLs of this type for the site), or universal. A fourth option — "URL pattern matching X" — would close the loop:

For our applications we have marked some issues as false positive. However, the endpoints of those issues are dynamic and contain a unique identifier e.g., /users/anonymous/<uuid>, meaning that Burp DAST sees it as a unique URL. This results in the reappearance of the issue that was previously marked as false positives in the next scan.
Rutger Hamulyak · AS Watson · ARR Tier 5 · Case 00347353 · Oct 2025

Net: the broad scope-clarity question (Tier 1 Carson 00307715) is resolved by the modal. The narrower dynamic-URL pattern problem (Babiel 00382654, AS Watson 00347353) remains as a genuine but more contained gap.

2.16 Re-test workflow — strongly validated by PM doc; my SF-only read missed the demand · PM Assertion 5 (validated)

Methodology correction

Earlier versions of this pack flagged retest as "no direct demand" because zero SF support tickets surfaced for it. That read was wrong. SF tickets capture pain about features that exist and are broken; they don't capture demand for features that don't exist yet. The PM doc, drawn from sales / discovery / renewal calls, surfaces the demand SF can't see — and it's substantial.

We give our developer access so that they can triage and fix issues, however the ability to re-test an issue is missing, this would be a big win for us.
Interpublic · $200–300k renewal opportunity · PM doc, Assertion 5
It's really important for those teams to be able to quickly rescan and verify whether the issue is resolved. Otherwise, they might have to wait a full day just to see if it's fixed, and even then, they can't be 100% sure.
Schwarz IT · $250k ACV renewal opportunity · PM doc, Assertion 5
If a vulnerability is fixed, I shouldn't have to rerun the whole scan to verify it. I should be able to simply re-scan the specific request-response to confirm whether the vulnerability is still present.
Visa · $90k ARR · PM doc, Assertion 5
Would like to hit re-test, as they need to start proving the metrics of the trending issues.
Telus Health · $183k ACV (year 3 ramp-up) · PM doc, Assertion 5

Four named customers, ~$700k+ in attached deal value (renewals + new ARR), all asking for the exact pattern v10 builds. Including a renewal opportunity at Interpublic (the same customer that endorsed the SLA dashboard in 2.2 and was lost in 2.1). Re-test isn't an over-investment — it's a renewal-driver.

The PM doc also frames the user need cleanly:

"I am a Security Engineer, trying to re-test specific vulnerabilities to confirm whether fixes worked. But I can only rerun full scans, not re-test selected issues. Which delays deployment, reduces confidence in remediation, and forces inefficient workarounds."

What this changes for the design work:

  • The state-transition bar (cyan / orange / grey) is the right level of feedback. Schwarz IT and Visa specifically describe the failure mode as "even then, they can't be 100% sure" — the bar's whole purpose is to remove that uncertainty.
  • Targeted retest at the request-response level is the demand (Visa). The bulk-retest verb in v10 covers the multi-issue case but the side-panel retest is the single-issue path — both are needed.
  • Speed matters. Schwarz IT's "wait a full day" sets the comparison; v10 should plausibly target retest as a minutes-not-hours workflow. Worth checking with engineering whether the back-end supports that.
  • Telus Health connects retest to metrics ("proving the metrics of the trending issues"). The retest events the prototype writes to the timeline are the data behind any "fixed-rate over time" chart — i.e. the SLA Fix Rate chart in Section 2.2 is downstream of retest data quality.

The methodology lesson stands as a Section 6 entry — see "What I learned about source mix."

2.17 Stat cards as filters — implicit interaction not surfaced by evidence

  • ✓ Verified risk — present (Accenture / Priceline ask, $344k attached value).
  • ✓ False Positive, Accepted Risk — present (today's defaults).
  • ✓ Fixed (unconfirmed) — present, distinct from confirmed-fixed-via-retest.

2.28 v11 site-tree navigator — the new primary navigation surface

The 340px left-side .site-panel is the most substantial new UI element across the whole prototype set, and I missed it in earlier reads. It's worth its own theme because it's the answer to several distinct customer needs:

  • Filter + Search across 172 sites — solves Tier 2 case 00357003 (UI Update Request — search field for large user / site lists) at a different surface.
  • Four-level nesting (estate / region / sub-folder / site) — matches the org-tree structure of large enterprise customers (Visa scale).
  • Selected-state visual on the active tree row — gives navigational sense-of-place when paginating across pages.
  • "Pentest group" classification sitting alongside geographic regions — acknowledges that grouping sites by team / engagement is as valid as grouping by region. Aligns with the System C Healthcare ask in 2.9 (third-party pen-test results as a managed category).
  • Persistent across all v11 pages — the tree stays open as you navigate, giving the user constant orientation. v10 doesn't have this; opening v10 means you've left the tree behind.

Open questions for the design team:

  • Persistence and collapse. 340px out of a 1440px laptop = 24% of the viewport. Combined with a 900px side panel (also opened from these pages), the tree + panel + small sliver of table is ~110% of the viewport. The tree probably needs a collapsed state (icon-only? hidden behind a hamburger?) — does v11 support this? Worth verifying.
  • Counts on tree rows. The folder rows show counts (Asia: 12). Are these issue counts, site counts, or scan counts? Different at different levels — disambiguating is worth thinking through.
  • Drag-to-reorganise, multi-select for bulk operations on the tree itself — both expected for an enterprise admin workflow but neither obvious in the prototype.

Working

Default page size 25 (or 50). 10 is a documents-app default, not a triage-app default.

  • Verified status is in v10 ("Verified risk" in the status select — see Section 2.24). Earlier "verify with the team" worry was unnecessary.
  • The 900px side panel is too wide for laptop users. Affects Scenarios 3 and 5. With v11's 340px tree adding to the takeover, this is even more pressing than first appeared.
  • Burp AI confidence — the most novel value — is partially hidden (Scenario 1, behind horizontal scroll) and partially incomplete (Scenario 3, advisory tab missing full evidence). The product's strongest differentiator deserves the design's strongest support.
  • Comments don't reach reports. A back-end issue, but it makes the Timeline tab a write-only audit log unless fixed (Scenario 3).
  • SLA Fix Rate chart hidden <1600px is a real defect for the persona this pack describes (Scenario 4).
  • v10 and v11 should feel like one product at two zoom levels, not two products. The site-tree should be available in v10 too (Scenario 5).

4What the evidence does not tell you

Adding Gong addressed two of the original blind spots and left three. Strikethroughs below mark items now (partially) covered.

  • Volume vs. importance. Jira integration generates the most cases because it's the most fragile integration; that doesn't necessarily make it the highest-leverage thing to redesign. SF over-indexes on broken things; Gong now provides a counter-balance by surfacing what customers volunteer they value, but still under-weights latent unmet needs.
  • Silent majority. Partially resolved by Gong — customers happy with today's product surface their satisfaction in calls, not in support tickets. Sections 2.18 (leaderboard) and 2.20 (WSAcad refs) are signals we'd have missed using SF alone.
  • Acquisition vs. retention. Resolved by PM doc — the PM brief explicitly attributes two lost deals (Interpublic $97k, Peach Payments $12k) and ~$700k of renewal opportunity to the gap this work addresses. Gong supplements with the "why customers chose DAST" framing (Section 2.30). The PM doc is the source that finally answers "is this an acquisition / churn issue?" with named-customer dollar attribution.
  • Recency bias. SF biased to last 24 months; the Gong synthesis is undated in the report I received, so the time-window is opaque. Worth checking with whoever generated the Gong report.
  • Sample. 26 SF cases drilled out of ~150 surfaced. Gong is a synthesis, so the underlying call sample size is unknown. Description-level signal in the SF long tail and the raw Gong transcripts both remain uncrawled.
  • Indirection in the Gong source. Gong's report is an AI synthesis of customer attitudes, not raw transcripts. Sentiment (e.g. "customers value X") is more reliable than specifics (e.g. exact phrasing or which customer said what). For decisions that hinge on a single customer's view, prefer SF cases or raw call recordings.

5Suggested follow-ups

  1. Confirm Bifrost / data-model alignment for stable issue identity. The PM doc explicitly flags this as a foundation requirement. v10/v11's row identity, Timeline events, and retest history all assume issues persist across scans with stable IDs. If the data model isn't ready, the prototype is designing on top of an unstable foundation. Highest priority — load-bearing for everything else in the pack.
  2. Unify v10 + v11 as one zoomable surface. Make the site-tree available in v10 too; opening v10 with no scope selected should be the natural "estate root" position; clicking a tree node zooms in. Removes the v10/v11 navigation gap (Scenario 5). PM Assertion 1 + Gong RBAC scoping align with this framing.
  3. Add URL-pattern scope option to the FP modal. The modal already resolves the broad scope question; what's missing is a fourth radio for URL-pattern marking ("all URLs matching /users/anonymous/* for this site"). Closes the Babiel + AS Watson dynamic-URL gap.
  4. Reconsider the bulk-modal default. Today it defaults to "this issue and all same-type for this site" — a broader scope than naive users may expect. Worth a usability test; "just this issue" may be a safer default.
  5. Coordinate triage states with the CI/CD pipeline gate. PM JTBD: "builds fail even when no genuine security risk remains". Verified-risk and Fixed-unconfirmed are both in v10's enum; the API the CI gate consumes needs to recognise them. Otherwise we make the pipeline problem worse, not better.
  6. Side panel + tree both open = no table. v11 makes the side-panel-width concern more pressing — tree (340px) + panel (900px) leaves ~200px of table on a 1440px laptop. Offer auto-collapse of the tree when the side panel is open, plus a 480px narrow-mode panel.
  7. Decide what happens to the vulnerable-sites leaderboard. Gong identifies it as something customers chose DAST for. v10 reframes the dashboard around SLA. Either preserve it elsewhere or take a deliberate decision to retire it. Don't lose it by accident.
  8. Audit / "proving what was checked". M&S and RXSense ($136k + $15k) describe issues vanishing without explicit Resolved transitions. Side-panel Timeline should record positive "verified clear" events, not silent disappearance. Small data-model addition with disproportionate audit-credibility return.
  9. Move SLA Fix Rate inside the laptop viewport. Either responsive-collapse to a KPI ribbon, or accept that the chart is desktop-only and add an explicit "view on a wider screen" affordance. Add MTTR explicitly (Interpublic Tier 1 ask).
  10. Tree-counts unit-of-measure label. "Asia: 12" reads as 12 issues but the data underneath suggests sites. Either standardise to issue-counts everywhere or label inline ("Asia · 3 sub-folders · 12 sites").
  11. Validate tree depth flexibility. Estate → region → sub-folder → site is one schema; real customers have hierarchies that don't fit it (BU, env, criticality). Worth confirming the tree degrades to two levels and extends to five.
  12. "Select all N matching filter" in the bulk-action bar. Standard pattern, missing from v10. Required for usability Scenario 2 to scale past one page.
  13. Verify WSAcad references in Advisory tab. Gong identifies these as customer-valued. Low-cost addition if missing.
  14. "Send to Burp Pro" workflow seam. Gong + PM doc both surface the DAST → Pro handoff. v10 currently has no action for this. PM doc explicitly cites this as Bifrost-related.
  15. Wire v11's RBAC. v11 currently scopes the view but not the access. Real enterprise deployments need access-controlled trees — the developer for Zenith Corp shouldn't see the rest of the estate.
  16. Comments → reports. Confirm with engineering whether timeline comments flow into exported reports (Manulife 00335249, Tier 1). If not, address before customers invest effort in commenting.
  17. Promote Burp AI output into the sticky region or fuse with Confidence (v9). The novel value is currently behind a horizontal scroll.
  18. Web/API filter. Decide whether to add as a first-class filter chip or push to tags-as-workaround. Customer ask is Tier 1.
  19. Linked-tickets persistence + auto-close. DKSH cases (00174085, 00184903). Side panel is the surface. Worth a conversation with engineering before committing the panel design.

6Workflow notes (for future runs of this pilot)

  • Salesforce schema is rich. Case.Enterprise_Components__c (DAST Components multipicklist) is the highest-leverage filter — values like Issue Management, Integrations - Issue Tracking, Site Management, Reporting map almost 1:1 to v10/v11 themes. Field is sparsely populated (only ~19 cases tagged "Issue Management" in 3 years), so keyword SOSL search across Subject/Description was needed to cover the long tail.
  • Idea / IdeaComment is a dead channel in this org (0 records returned). Don't waste a query on it.
  • Tier-weighting matters. Tier 1 / Tier 3 case asks should get priority over the Tier 4 / Tier 5 long tail. The ARR_Tier__c field makes this trivial.
  • Description text is where the evidence lives — Subject lines are too sparse. Always pull Description on the most relevant cases.
  • Multipicklist gotcha: can't GROUP BY Enterprise_Components__c, only filter via INCLUDES (...).
  • Cost guidance: ~9 SOSL queries + 2 batch SOQL for descriptions covered the surface adequately for a 17-theme design + 5-scenario walkthrough. A deeper pass would also crawl CaseComment.CommentBody for follow-up dialogue, and join cases to Account for industry-vertical patterns.
  • Watch for repeat customers. Carson opened both 00307715 (status scope) and 00322295 (confidence filter) — same workflow, two unanswered halves. Joining cases by Contact / Account surfaces these threads and tells you which customers have multiple unaddressed asks. Tier 1 + repeat = highest priority.
  • Pair the evidence pass with a usability walkthrough every time. Customer cases tell you what users say; the walkthrough tells you what users will hit when they actually use the design. The two views together caught issues that neither would catch alone — e.g. the SLA Fix Rate chart's viewport-size problem is a usability finding that no customer case could have surfaced, because no customer has used the prototype yet.
  • SF + Gong is the right two-source minimum. SF tells you what's broken; Gong tells you what's loved. Designs that look fine through one lens often look different through the other — e.g. the vulnerable-sites leaderboard is invisible in SF (no one opens a ticket about a feature that works) but central in Gong (customers volunteer that they chose DAST for it). Run both for any design that touches a customer's daily workflow.
  • Gong's AI synthesis is a starting point, not an end-point. The summary is useful for direction-setting but flattens detail. For decisions that hinge on specifics — exact phrasing, which customer said what, time period — go back to the raw transcripts or pull the relevant calls directly. The synthesis lacks dates, customer attribution, and cohort sizes.
  • Watch for what only one source surfaces. The most strategically important finding in this pack — Section 2.30, "what's at stake commercially" — needs Gong + PM doc together (neither alone tells you both why customers chose DAST and what dollar value is on the table). The most consequential design correction in the pack — Section 2.16, retest demand — appears only in PM doc (zero SF tickets).
  • "No SF tickets" doesn't mean "no demand." Customers don't open support tickets about features that don't exist yet — they ask about them in sales / discovery / renewal conversations, which land in PM discovery briefs and Gong. My initial Section 2.16 finding ("retest has no direct demand") was a methodology error that the PM doc corrected: 4 named customers, ~$700k of attached deal value. Whenever a feature in the prototype shows zero SF tickets, treat that as a flag to look in PM/Gong before concluding "no demand." It often means "demand exists but in a different conversation."
  • Read the actual prototype, all of it. Two of the most prominent gaps in earlier versions of this pack — "scope ambiguity" and "verify Verified status exists" — were corrected by reading the bulk-action modals (which I'd previously skipped) and the v11 architecture (which I'd misread as "single-site"). Source-mining is necessary; complete prototype reading is also necessary. Don't audit a design from its summary or its index — read it end-to-end. Same lesson as the source mix: each surface tells you something the others miss.
